# Tipos de Proyecto Dinámicos y Gantt — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the Rails backend + basic admin/CRUD UI for a project-management platform where project types (and their custom fields + Gantt subprocess stages) are defined by an admin at runtime, starting with the "Instalaciones" type.

**Architecture:** Rails app on Postgres. `ProjectType` owns `FieldDefinition` (metadata describing custom fields, rendered dynamically) and `StageTemplate` (fixed subprocess sequence). `Project` stores custom field values in a `jsonb` column validated against its type's `FieldDefinition`s. On creation, a `Project` copies its type's `StageTemplate`s into `ProjectStage` rows — the actual Gantt bars, each with its own dates/progress/responsible. `Installer` is a small growing catalog referenced by `reference`-type fields.

**Tech Stack:** Ruby 3.4.4, Rails 7.2.3, PostgreSQL (via Docker for local dev), Minitest (Rails default — no RSpec, avoids an extra dependency per ponytail).

## Global Constraints

- Admin defines new project types (fields + stages) from the UI — no code deploy required for a new type. (spec: "Requisito clave")
- `custom_fields` on `Project` is a single `jsonb` column; no EAV, no per-type tables. (spec: "Enfoque elegido")
- `ProjectStage` rows are a frozen copy of `StageTemplate` at creation time — editing a template never retroactively changes existing projects. (spec: "Casos borde")
- `data_type` for `FieldDefinition` is one of: `text`, `date`, `percent`, `reference`. (spec: "Modelo de datos")
- Gantt JS visualization library is explicitly out of scope for this plan — only the data + a plain table view. (spec: "Fuera de alcance")
- Auth/roles are out of scope — `assigned_user_id` on `ProjectStage` is a bare integer column, no `User` model/FK yet. (spec: "Fuera de alcance")
- Keep files under 500 lines (project CLAUDE.md).
- No files in repo root — app code lives under Rails' standard `app/`, `db/`, `test/` dirs (project CLAUDE.md).

```ruby
class Project < ApplicationRecord
  belongs_to :project_type

  validates :name, presence: true
  validate :custom_fields_match_definitions

  private

  def custom_fields_match_definitions
    project_type.field_definitions.each do |field|
      value = custom_fields[field.key]
      next if value.nil?

      case field.data_type
      when "text"
        errors.add(:custom_fields, "#{field.label} debe ser texto") unless value.is_a?(String)
      when "date"
        errors.add(:custom_fields, "#{field.label} debe ser una fecha válida") unless valid_date?(value)
      when "percent"
        errors.add(:custom_fields, "#{field.label} debe ser un porcentaje entre 0 y 100") unless valid_percent?(value)
      when "reference"
        errors.add(:custom_fields, "#{field.label} debe referenciar un registro existente") unless valid_reference?(field, value)
      end
    end
  end

  def valid_date?(value)
    Date.parse(value.to_s)
    true
  rescue ArgumentError, TypeError
    false
  end

  def valid_percent?(value)
    Float(value) rescue false
    Float(value).between?(0, 100)
  rescue ArgumentError, TypeError
    false
  end

  def valid_reference?(field, value)
    field.reference_table.classify.constantize.exists?(id: value)
  rescue NameError
    false
  end
end
```
