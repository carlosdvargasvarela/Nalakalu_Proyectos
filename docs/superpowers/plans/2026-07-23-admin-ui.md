# Pulido de las vistas de Administración, fix de borrado roto — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix a real bug — deleting a field definition or stage template from Administración currently 404s because the "Eliminar" links rely on `data-turbo-method`, and this app never loads Turbo's JS — and group "Campos"/"Subprocesos" into cards matching the visual pattern already used on the project detail page.

**Architecture:** Pure view-layer change. No controller/route/model changes — `Admin::FieldDefinitionsController#destroy` and `Admin::StageTemplatesController#destroy` already work correctly (proven by existing controller tests that call `delete` directly); only the *link* that was supposed to trigger them was broken. Switches those links to `button_to` (the pattern already working for installers), adds a native `confirm()` safety prompt via `onsubmit` to all three delete forms (field definitions, stage templates, installers — none had one before, since the Turbo/rails-ujs-flavored confirm attributes would have been equally inert).

**Tech Stack:** Ruby on Rails, Minitest. No new JS, no Turbo, no rails-ujs — plain HTML `onsubmit="return confirm(...)"`, which needs no framework.

## Global Constraints

- No new gems, no Turbo/Stimulus wiring — this app deliberately has none loaded in the layout.
- `confirm()` must be wired via the `<form>`'s `onsubmit` (which `button_to`'s `form:` option can set), not the button's `onclick` — only cancelling the form's `submit` event actually stops the request; a button-level `onclick` returning `false` does not prevent the form it's inside from submitting.
